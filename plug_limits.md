# Динамические ограничения на создание дочерних объектов

## Контекст

В модели плагинов `plugin.json` описывает базовую топологию объектов: какие типы объектов могут быть дочерними для каких родительских типов.

Пример:

```json
{
  "type": "SENSOR",
  "parent": "CONTROLLER"
}
```

Это означает, что объект типа `SENSOR` в принципе может быть создан под объектом типа `CONTROLLER`.

По умолчанию ограничений на количество дочерних объектов нет. Плагин часто не знает фиксированный лимит на этапе описания манифеста: ограничение может зависеть от модели устройства, лицензии, текущего состояния, внешней системы или уже созданных дочерних объектов.

Поэтому ограничения вида «для этого конкретного `CONTROLLER` можно создать не более двух `SENSOR`» лучше проверять динамически через RPC-вызов к плагину.

## Предлагаемый RPC

```proto
rpc CanCreateChild(CanCreateChildRequest) returns (CanCreateChildResponse);

message CanCreateChildRequest {
  ObjectReference parent = 1;
  ObjectType child_type = 2;
  google.protobuf.Struct child_params = 3;
}

message CanCreateChildResponse {
  bool allowed = 1;
  string reason_code = 2;
  string reason_message = 3;
  optional uint32 current_count = 4;
  optional uint32 max_count = 5;
}
```

## Поток работы платформы

Когда оператор создаёт дочерний объект:

1. Платформа проверяет статическую топологию из `plugin.json`.
2. Если топология допускает такой дочерний тип, платформа вызывает `Plugin.CanCreateChild`.
3. Если `allowed = true`, платформа разрешает создание объекта.
4. Если `allowed = false`, платформа отклоняет создание и показывает оператору причину из `reason_message`.

Пример: `CONTROLLER` в целом может иметь дочерние `SENSOR`, но конкретный контроллер разрешает не более двух датчиков. Тогда плагин может вернуть:

```proto
CanCreateChildResponse {
  allowed: false
  reason_code: "LIMIT_REACHED"
  reason_message: "Controller already has the maximum number of sensors"
  current_count: 2
  max_count: 2
}
```

## Почему не только `plugin.json`

Статическое описание ограничения в манифесте, например:

```json
{
  "type": "CONTROLLER",
  "child_constraints": {
    "SENSOR": { "max": 2 }
  }
}
```

полезно только тогда, когда лимит универсален и заранее известен для всех объектов данного типа.

В этой модели плагинов большинство ограничений ожидаются динамическими или опциональными. Если хранить такие лимиты только в `plugin.json`, придётся заранее кодировать ограничения, которые на самом деле применимы только к отдельным устройствам, лицензиям или runtime-состояниям.

## Правило

`plugin.json` определяет, что структурно возможно.

`CanCreateChild` определяет, что разрешено прямо сейчас для конкретного родительского объекта.
